# Monsters 設計書

バトルえんぴつ風の2人対戦ゲーム。プレイヤーが作ったモンスターがサイコロの出目に応じて行動し、HPを削り合う。

## 1. 概要

| 項目 | 内容 |
|------|------|
| ゲーム名 | Monsters |
| ジャンル | ターン制サイコロバトル（バトルえんぴつ風） |
| プレイ人数 | 1〜2人 |
| 勝敗 | HP制（先に相手のHPを0にしたら勝ち） |
| プラットフォーム | PC（ブラウザ） |
| ルールモード | フリービルド / バランスビルド の2種（キャラ設計の制約） |
| 対戦形態 | ローカル2人 / CPU対戦 / 練習対戦 / オンライン2人（フェーズ2） |
| 開発段階 | フェーズ1: ローカル1台で交互プレイ → フェーズ2: オンライン対戦 |
| ホスティング | Render 無料枠（フェーズ2。keep-aliveなし・スリープ許容） |

設計方針として、**ゲームロジック（ルール計算）をUIや通信から完全に分離**する。これによりローカル版とオンライン版でロジックを共有でき、サーバー側でも同じ判定コードを使える。

## 2. ルールモード

ゲーム開始時にどちらかのモードを選ぶ。バトル中の戦闘ロジックは両モードで完全に共通で、**違いはモンスター作成画面の制約だけ**。

### モードA: フリービルド（FREE BUILD）
- 制限なし。HPも各面の効果量も自由に設定できる
- 身内で遊ぶ・好きなモンスターを再現する・お祭り対戦向け
- バランスは保証しない（ロマン重視）

### モードB: バランスビルド（BALANCE BUILD）
- **総ポイント（予算）の中でHPと全行動を購入する**コスト制
- 同じ予算で組むので対人戦が公平になる。オンライン対戦のデフォルト想定
- 予算超過があるとバトル開始できない（余りは許容）

## 3. 対戦形態

ルールモード（キャラ設計の制約）とは別軸の、**誰がどの手番を操作するか**の選択。戦闘ロジックは全形態で完全に共通で、**違いは「相手の手番を誰が振るか」だけ**。出目で行動が自動確定する設計なので、プレイヤーの介入は「ダイスを振る」操作のみ。

| 形態 | 操作 | 用途 |
|------|------|------|
| ローカル2人 | 両手番とも人間（交互に1台で） | 友達と同じ画面で対戦 |
| CPU対戦 | 自分=人間 / 相手=CPU | ひとりで対戦を楽しむ・作ったキャラの強さを試す |
| 練習対戦 | 両手番とも自分が操作 | 2体の相性確認・バランス検証（ホットシートを1人で回す） |
| オンライン2人 | 各手番を別端末の人間 | フェーズ2。離れた相手と対戦 |

**CPUの仕様**: このゲームは出目で行動が確定し、プレイヤーに出目以外の選択余地がない。よってCPUに戦略・賢さの概念は不要で、**自分の手番が来たら自動でダイスを振るだけ**。実装は「手番がCPU側なら rollDice→applyAction を自動実行する」処理を1つ挟むのみで、戦闘判定コードには一切手を入れない。将来も出目以外の介入要素は追加しない方針なので、この単純さは恒久的に保たれる。

練習対戦はローカル2人と中身がほぼ同じ（両手番を同じ人間が操作）。CPU対戦との差は「相手手番の自動化の有無」だけ。

### CPU対戦の相手モンスターの供給源
3つを併用でき、CPU対戦の準備画面で選べる。

- **保存済みから選ぶ** — 自分や友達が作って読み込んだ `.monst` を相手にする
- **プリセット同梱** — 最初から数体入れておく（チュートリアル的な相手・腕試し用）
- **ランダム自動生成** — その場で1体でっち上げる。手軽さ担当。相手がいなくてもワンタップでCPU戦が始まる

自動生成は「AIがキャラを考える」のではなく、**固定された Character 構造（name, maxHp, actions[6]）の各フィールドを ruleConfig の枠内でランダムに埋めるだけ**。
- フリービルド用: 常識的な範囲でランダム（例: HP 50〜200、power 1〜20）。極端・全ハズレ面を避ける軽い下限を設ける
- バランスビルド用: ランダムに組んで `validateCharacter` を通し、予算超過なら作り直す（数回リトライ）。既存のコスト計算をそのまま使え、相性が良い
- 名前・画像はプレースホルダ（"CPU"＋ランダム接尾辞、デフォルト画像）

## 4. 行動の効果と「マイナス効果」

各面は type と power を持つ。**power は負の値も設定でき、自分が不利になる（=デメリットの）行動を作れる。** マイナス効果はコストが下がり（ポイントが浮き）、その浮いたポイントで他の面を強化できる、というハイリスク・ハイリターンのビルドが組める。

| type | power>0 の効果 | power<0 の効果（マイナス） |
|------|----------------|----------------------------|
| attack | 相手に power ダメージ | **自分に \|power\| ダメージ**（暴発・自爆面） |
| heal | 自分を power 回復 | **自分が \|power\| ダメージ**（毒・反動面） |
| guard | 次の被ダメを power 軽減 | **次の被ダメが \|power\| 増加**（無防備面） |
| special | power×2 ダメージ | **自分に \|power\|×2 ダメージ**（大暴発面） |
| transform | 別フォームに変身（power不使用、targetForm を参照） | — |
| none | 効果なし | — |

> 例: 「6の目で相手に40ダメージ、ただし1の目で自分に20ダメージ」のような、博打性の高いモンスターが作れる。マイナス面でコストが浮くので、その分を大ダメージ面に回せる。

変身（transform）は別フォームへ丸ごと切り替わる特別な行動。詳細は次章。

## 5. 変身（フォーム）機能

特定の出目で別キャラクター（フォーム）に丸ごと変身する。強化変身も弱体変身も作れる。両モードで使用可。

### モデル: モンスターは複数フォームの集合
変身を入れると「1モンスター = 複数の Character（フォーム）」になる。

```
Monster {
  forms: Character[]        // forms[0] が初期フォーム
}
```
- 各フォームの面に type:"transform" を置き、`targetForm`（forms内のindex）で変身先を指す
- 多段変身: 変身先フォームの面がさらに別フォームを指せる（A→B→C…）
- 元に戻る: 変身先の面が初期フォームを指せば「戻る」も表現できる（専用機能は不要）
- **多段の上限なし**。循環（A→B→A 等）も許容する（下記コスト計算で安全に処理）

### HPの扱い: 割合維持
変身時、**HP割合を引き継ぐ**。変身前HP/変身前maxHp の割合を、変身先の maxHp に掛けた値を新HPとする。
```
新HP = round(変身先.maxHp × (現HP / 現フォーム.maxHp))
```
- 「変身＝全回復」の悪用を防ぐ
- 低HPフォームへ変身すると実数HPが減る＝弱体変身のリスクにもなる
- guardValue は変身でリセット（フォームが変わるので持ち越さない）

### コスト計算（割引・循環安全）
変身先フォームは「確率的にしか使えない」ため割り引いて合算する。初期フォームは毎ターン使うので満額。

```
モンスター総コスト:
  reachable = forms[0] から transform 面をたどって到達できる全フォーム集合
              （Set で訪問管理＝循環があっても各フォーム1回だけ）
  total = calcCharacterCost(forms[0])              // 初期フォームは満額
  for each f in reachable (f ≠ forms[0]):
      total += calcCharacterCost(f) × FORM_DISCOUNT  // 変身先は割引
  → total を BUDGET と比較（balance時）

FORM_DISCOUNT = 0.5（まず一律。対戦して強弱を見て調整）
```
- 循環安全装置: 到達可能フォームを Set で集めるので、輪になっていても各フォームを1回しか数えない＝無限ループしない
- transform 面そのもの（面のコスト）は 0 とする（変身先フォームのコストで既に課金しているため二重取りを避ける）

## 6. コスト制（バランスビルドの中身）

総予算 `BUDGET`（初期値 100pt 想定、調整可）を以下に配分する。

### HPのコスト
```
HPコスト = floor(maxHp / HP_PER_POINT)
```
- 例: `HP_PER_POINT = 5` なら HP100 = 20pt
- HPは下限・上限を設ける（例: 50〜200）

### 行動（各面）のコスト ※マイナス対応
power の符号でプラス面（自分に有利）かマイナス面（自分に不利）かが決まる。

```
power >= 0 の面:
  面コスト = TYPE_BASE_COST[type] + power × POWER_COST[type]

power < 0 の面（デメリット面）:
  面コスト = -( |power| × POWER_REFUND[type] )   // 負のコスト＝ポイントが戻る（割引）
```

係数（初期案、要バランス調整）:

| type | 効果 | TYPE_BASE_COST | POWER_COST | POWER_REFUND（マイナス時の払い戻し） |
|------|------|----------------|------------|--------------------------------------|
| attack | 攻撃 | 0 | 1.0 | 0.6 |
| heal | 回復/反動 | 1 | 1.0 | 0.6 |
| guard | ガード/無防備 | 1 | 0.8 | 0.5 |
| special | クリティカル/大暴発 | 3 | 1.5 | 0.8 |
| none | ハズレ面 | 0 | 0 | — |

> 払い戻し係数（REFUND）はコスト係数より低く設定する。等価にすると「マイナス面を盛って浮かせる」が無限に得になり崩壊するため、**リスクを取っても完全には割に合わない**ようにしてバランスを取る。

### 合計コストの制約
```
総コスト = HPコスト + Σ(各面コスト)   // 各面コストはマイナスもあり得る
バランスビルド成立条件: 総コスト <= BUDGET かつ 総コスト >= 0（下限0でクランプ）
```

作成画面では「残りポイント」をリアルタイム表示。超過したら赤字＋開始ボタン無効。マイナス面で浮いた分が残りに加算される。

## 7. 技術スタック

### フェーズ1（ローカル）
- HTML + CSS + Vanilla JS（ビルド不要）／ または Vite + TypeScript（推奨）
- 画像はブラウザ内で `FileReader` により Base64 / Blob URL で保持（縮小・デフォルトの詳細は8章「画像の扱い」）

### フェーズ2（オンライン）
- 通信: WebSocket
- サーバー: Node.js + `ws`
- **ホスティング: Render 無料枠で確定**（2026年時点、カード登録不要で常駐WebSocketを無料で動かせる唯一の現実的選択。Fly.ioは無料枠廃止、Railwayは恒久無料枠なし、Vercel/Netlifyのサーバーレスは常時接続のWSに不向き）
  - フロント（静的サイト）と WSサーバー（Webサービス）を両方 Render の無料枠で配信
  - **keep-alive（定期ping）は行わない。** 個人で時々遊ぶ用途では、誰も遊んでいない間もサーバーを起こし続けるのはリソースの無駄遣いであり、無料枠の趣旨にも反する。スリープは素直に受け入れる
  - コールドスタートの挙動: 15分アクセスがないとサーバーがスリープし、次の接続で起動に30〜50秒かかる。対戦が始まれば起き続けるので、待つのは「久しぶりに開いた最初の1回」だけ
  - 対処は演出のみ: 接続中に「サーバーを起こしています…（最大1分ほどかかります）」というローディング表示を出す。リソースを消費せず、ユーザーに状況を伝えるだけの行儀のよい対応

## 8. 画像の扱い

各フォームのキャラ画像のアップロード・縮小・デフォルトの仕様。

### 許可形式
png / jpeg / webp / gif を許可。
- **gif は静止画として扱う**。縮小処理（canvas経由）でアニメは保持されず1コマ目の静止画になる。アニメ保持はフレーム分解・再構成が必要で縮小と両立しづらく実装が重いため、まずは静止画化で割り切る（将来必要なら専用対応を追加）

### サイズ: 上限超過時のみ縮小
**長辺1024px を上限**とする。
- 1024px以下の画像はそのまま使う（縮小しない＝原画質を保つ）。常識的なキャラ画像はほぼ無加工
- 1024pxを超える画像のみ、アスペクト比を保って長辺1024pxに収まるよう canvas で自動縮小
- 補助として1枚あたりのファイルサイズ上限も置く（例: 元ファイル数MBまで）。極端な巨大ファイルでの事故防止
- 縮小後の画像を保存・表示・zip同梱・オンライン送信のすべてで一貫使用。これにより `.monst` もオンライン通信も過度に重くならない

### デフォルト画像: 頭文字アバター
画像未設定でもバトル可能にするフォールバック。
- アセット画像を持たず、**フォーム名の頭文字＋名前から決定的に生成した背景色**をその場で描画（canvas or CSS）
- 同じ名前は常に同じ色になり識別しやすい。変身フォームごとに違う頭文字・色が自然に出る
- 名前も空なら「？」等のプレースホルダ文字を使う

## 9. データモデル

### Action（サイコロ1面の行動）
```
Action {
  id: string
  type: "attack" | "heal" | "guard" | "special" | "transform" | "none"
  label: string        // 表示名（例: "つよいパンチ" / "自爆" / "覚醒"）
  power: number        // 効果量。負の値でデメリット面。none/transform は 0
  targetForm?: number  // type==="transform" のときのみ。変身先フォームの index
}
```

効果の適用ルール（power の符号で分岐）:

| type | power>0 | power<0 |
|------|---------|---------|
| attack | 相手に power ダメージ（相手ガード中は power-guardValue、最低0） | 自分に \|power\| ダメージ |
| heal | 自分を power 回復（maxHp上限） | 自分に \|power\| ダメージ |
| guard | 自分の guardValue = power（次の被ダメ軽減、1回分保持） | 自分の guardValue = power（負値＝次の被ダメ増加） |
| special | 相手に power×2 ダメージ（ガード貫通は要決定） | 自分に \|power\|×2 ダメージ |
| transform | 自分が forms[targetForm] に変身。HP割合維持・guardValueリセット | — |
| none | 効果なし | — |

### Character（フォーム）
```
Character {
  name: string
  imageUrl: string         // Blob URL or Base64
  maxHp: number
  actions: Action[6]       // サイコロの目1〜6に対応（必ず6個）
}
```

### Monster（1体のモンスター＝フォーム集合）
```
Monster {
  forms: Character[]       // forms[0] が初期フォーム。変身なしなら要素1つ
}
```
変身を使わないモンスターは forms が1要素なだけで、構造は共通。

### RuleConfig（モード設定）
```
RuleConfig {
  mode: "free" | "balance"
  budget: number
  hpPerPoint: number
  hpMin: number
  hpMax: number
  typeBaseCost: Record<type, number>
  powerCost: Record<type, number>
  powerRefund: Record<type, number>   // マイナス面の払い戻し係数
}
```

### GameState（対戦中の状態）
```
GameState {
  players: [PlayerState, PlayerState]
  matchType: "local" | "cpu" | "practice" | "online"
  controllers: ["human" | "cpu", "human" | "cpu"]
    // 各手番を誰が操作するか。
    // local/practice = ["human","human"]、cpu = ["human","cpu"]、online = ["human","human"]（別端末）
  turn: 0 | 1
  phase: "initiative" | "rolling" | "resolving" | "finished"
  winner: 0 | 1 | null
  log: LogEntry[]
}

PlayerState {
  monster: Monster          // 全フォームを保持
  currentForm: number       // 現在のフォーム index（初期=0）。変身でここが変わる
  hp: number
  guardValue: number        // 正=軽減、負=被ダメ増加。変身でリセット
  // 現在のCharacterは monster.forms[currentForm] で取得
}

LogEntry {
  turn: number
  playerIndex: 0 | 1
  diceRoll: 1..6
  action: Action
  result: string            // "自分に20ダメージ(自爆)" 等
}
```

## 10. コア関数 API

UI・通信から独立した純粋関数として実装する。

```
// --- ルール/コスト層 ---
calcActionCost(action, ruleConfig): number          // 負の値もあり得る。transform/none は 0
calcCharacterCost(character, ruleConfig): number     // 1フォーム分: HP+全面合計（下限0クランプ）
calcMonsterCost(monster, ruleConfig): number
  // forms[0] 満額 + 到達可能な変身先を FORM_DISCOUNT 倍で合算。
  // 到達フォームは Set で集め循環安全
validateMonster(monster, ruleConfig):
  { ok: boolean, totalCost: number, remaining: number, errors: string[] }
generateRandomMonster(ruleConfig): Monster
  // ruleConfig の枠内でランダム生成。balance時は validateMonster を通し超過なら作り直す。
  // CPU対戦の相手用（まずは単一フォーム中心で可）

// --- 画像処理層 ---
processImage(file): Promise<Blob>
  // png/jpeg/webp/gif を受け、長辺1024px超なら canvas でアスペクト比維持縮小（以下なら原画のまま）。
  // gif はこの時点で静止画化される。結果を保存・表示・送信に一貫使用
defaultAvatar(name): string
  // 画像未設定時。名前の頭文字＋名前から決定的に生成した背景色のアバターを描画して返す

// --- 書き出し/読み込み層（JSZip 使用）---
exportMonster(monster): Blob
  // .monst（zip）を生成。全フォームの monster.json と各フォームの画像を同梱
sanitizeFileName(name): string
  // ファイル名用に安全化。使用不可文字を置換、長さ切り詰め、空なら "monster"
downloadMonster(monster)
  // exportMonster + sanitizeFileName で `{forms[0]の安全化名}.monst` としてダウンロード
importMonster(file): Promise<Monster>
  // .monst（zip）を読み、全フォームと画像を復元して Monster を返す

// --- 戦闘層（モード非依存・共通）---
createGame(monsterA, monsterB, matchType): GameState // controllers設定、currentForm=0、phase="initiative"
rollInitiative(state): GameState
  // 両者がダイスを1回振り、出目が大きい方を先攻(turn)に設定。
  // 同値なら割れるまで内部で振り直す。結果を log とともに state に反映し phase="rolling" へ
rollDice(): number                                  // 1〜6
applyAction(state, playerIndex, diceRoll): GameState
  // 現フォーム forms[currentForm] の出目の面を引く。
  // transform面なら HP割合維持で変身（currentForm更新・guardValueリセット）。
  // それ以外は power符号で自他どちらに作用かを判定して適用
checkWinner(state): 0 | 1 | null
isCurrentTurnCpu(state): boolean                     // controllers[turn] === "cpu"
  // UI層がこれを見て、CPU手番なら自動で rollDice→applyAction を実行（少し間を置いて演出）。
  // 戦闘判定コードはCPUを意識しない
```

`validateMonster` がモード差を吸収する唯一の場所（free は常に ok=true）。戦闘ロジックはモードを知らない。`applyAction` は出目の面の type を見て、transform なら変身、それ以外は power の符号で自分/相手どちらに効果を向けるかを判定する。変身は HP割合維持・guardValueリセットで currentForm を切り替えるだけ。

## 11. モンスターの書き出し・読み込み

作ったモンスターをファイルとして外に出し、保存・共有・読み込みできる。対戦相手がいなくてもキャラを蓄積でき、後述のソロモードとも噛み合う。

### 形式: `.monst`（実体は zip）
zip ライブラリ **JSZip** を使用（ブラウザでzipを生成・解凍する定番。Vanilla JS でも導入可）。中身は構造データのJSONと画像を別々に同梱する。Base64で画像を文字列化しないので、JSONは読みやすく、画像は劣化・肥大化なしで運べる。

```
{モンスター名}.monst  (zip)
├─ monster.json     # forms配列（各フォームの名前・maxHp・actions[6]・画像参照名）
├─ form0.png        # 初期フォームの画像
├─ form1.png        # 変身先フォームの画像（フォーム数ぶん）
└─ ...
```
- monster.json は Monster 構造（forms[]）をそのまま格納。各フォームの画像は参照名（form0.png 等）で持つ
- 変身なしのモンスターは forms が1要素＝画像1枚なだけで、同じ構造

- まずは **1ファイル = 1モンスター**（複数まとめてのバックアップは将来拡張）
- 書き出し・読み込みの両対応

### ファイル名ルール
ダウンロード名は `{サニタイズした初期フォーム名}.monst`。`monster.zip (1)(2)...` 地獄を避け、フォルダで一目で識別できるようにする。

- ファイル名に使えない文字（`/ \ : * ? " < > |` 等）は `_` に置換
- 長すぎる名前は適度に切り詰め
- 名前が空なら `monster` を既定名に
- **元のモンスター名は monster.json 内に正確に保持**（表示名は損なわない。安全化はファイル名用の別処理）

## 12. 画面構成

1. **タイトル** — 「あそぶ」
2. **対戦形態選択** — ローカル2人 / CPU対戦 / 練習対戦 / オンライン2人（フェーズ2）
3. **モード選択** — フリービルド / バランスビルド。バランス時は予算表示
4. **モンスター作成** — 名前、画像アップロード（プレビュー）、HP、6面の type・名前・power を設定。power はマイナス入力可。**変身を使う場合はフォームを追加し、各フォームごとに画像・HP・6面を設定。transform面では変身先フォームを選ぶ**。バランスビルドでは上部に「残り XX pt」を常時表示（変身先は割引で加算）し、超過で開始無効。マイナス面はUI上で色分け（赤＝デメリット）して誤設定を防ぐ。CPU対戦では相手を「保存済み / プリセット / ランダム生成」から選ぶ。練習対戦では2体とも自分が用意
5. **先攻決定** — バトル開始時、両者がダイスを1回振り出目の大きい方が先攻。同値なら割れるまで振り直し。演出として見せる
6. **バトル** — 左右に両モンスター（現フォームの画像・名前・HPバー）、中央サイコロ、下部に行動ログ、手番表示、「サイコロを振る」。自爆など自傷時はログ・演出で明示。**変身時は画像・名前を切り替え、変身演出とログを出す（HPは割合維持）**。CPU手番では `isCurrentTurnCpu` を見て、少し間を置いてから自動でダイスを振る
7. **結果** — 勝者表示、「もう一度」「編集に戻る」

## 14. フェーズ2 オンライン化メモ

- `applyAction` 呼び出しを「サーバーへ手番送信 → サーバーが権威ある状態を計算 → 全クライアントへ配信」に置換
- サーバーは `game.js` と `rules.js` をそのまま流用（同じ判定・同じコスト検証で不正防止）
- **バランスビルドはサーバー側で必ず validateCharacter を再検証**（クライアントの予算チェックは信用しない。マイナス面の払い戻し悪用も防ぐ）
- マッチング: ルームコード方式。部屋作成時にモードと予算を確定し参加者に強制
- ダイス目はサーバー生成（チート防止）
- 試合終了時にサーバー側の部屋を破棄してメモリ解放（無料枠の小さいRAMでも複数部屋をさばける）

### デプロイ構成（Render 無料枠）
- フロント = Render の静的サイト、WSサーバー = Render の Webサービス
- keep-alive は入れない。スリープは許容する方針
- WSサーバーに軽量な `/health` エンドポイントだけ用意しておく（死活確認・将来必要になった時用。常時pingはしない）
- クライアントの接続フロー:
  1. WS接続を試みると同時に「サーバーを起こしています…（最大1分ほど）」のローディングを表示
  2. コールドスタート中は接続が確立するまで数十秒かかり得るので、タイムアウトを長め（例: 60〜90秒）に設定し、その間リトライ
  3. 接続確立したらローディングを消してロビー/対戦画面へ
- 一度繋がれば対戦中はサーバーが起き続けるので、以降のやり取りは即時

## 15. 決定事項

- **先攻・後攻**: バトル開始時に両者がダイスを1回振り、出目の大きい方が先攻。同値なら割れるまで振り直し。この出目は手番順を決めるだけで勝敗には影響しない
- **モンスター書き出し**: `.monst`（zip）形式。monster.json（forms配列）と各フォームの画像を同梱し JSZip で生成/解凍。1ファイル1モンスター、書き出し/読み込み両対応。ファイル名は `{サニタイズした初期フォーム名}.monst`（不可文字置換・長さ切り詰め・空なら monster、元の名前はJSON内に保持）
- **対戦形態**: ローカル2人 / CPU対戦 / 練習対戦（自分で両手番）/ オンライン2人（フェーズ2）の4種。戦闘ロジックは共通で、違いは「相手手番を誰が振るか」だけ。CPUは出目以外の選択余地がないため賢さ不要＝自手番で自動でダイスを振るのみ。将来も出目以外の介入要素は追加しない
- **CPU相手の供給源**: 保存済み / プリセット同梱 / ランダム自動生成 の3つを併用。自動生成は固定Character構造を ruleConfig 枠内でランダムに埋め、balance時は validateMonster で予算チェックして作り直す
- **変身（フォーム）機能**: 1モンスター=複数フォーム（Character）の集合。transform面が出ると forms[targetForm] に変身。HPは割合維持、guardValueリセット。多段変身・循環を許容。両モードで使用可。コストは初期フォーム満額＋変身先を FORM_DISCOUNT=0.5 で合算（到達フォームをSetで集め循環安全、transform面自体のコストは0）
- **画像**: png/jpeg/webp/gif を許可（gifは静止画化）。長辺1024pxを上限とし、超過分のみ canvas で縮小（以下は原画質保持）。縮小後を保存・表示・送信に一貫使用。未設定時は名前の頭文字＋名前由来の背景色のアバターを生成

## 16. 未確定・要調整パラメータ

- BUDGET 初期値（100pt案）と各係数・払い戻し係数のバランス
